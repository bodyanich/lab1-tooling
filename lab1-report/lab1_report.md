# Звіт до лабораторної роботи №1

## Тема

**Налаштування професійного інструментарію розробки на Go**

## Мета роботи

Метою лабораторної роботи є ознайомлення з базовим інструментарієм розробника Go, який використовується для підтримки якості коду, автоматизації типових дій і перевірки коректності програмної логіки. У межах роботи було налаштовано структуру Go-проєкту, створено unit-тести, підключено статичний аналіз коду через `golangci-lint`, описано типові команди в `Makefile`, а також підготовлено репозиторій для роботи з Git та GitHub.

## Використане програмне забезпечення

- Go SDK версії 1.21 або новішої;
- Visual Studio Code;
- розширення Go для VS Code;
- Git;
- GitHub;
- `golangci-lint`;
- `make`;
- PowerShell.

## Хід виконання роботи

### 1. Ініціалізація Go-модуля

Було створено нову директорію проєкту та ініціалізовано Go-модуль:

```powershell
go mod init github.com/bborodiy41/lab1-tooling
```

У результаті було створено файл `go.mod`.

### 2. Реалізація програмної логіки

У файлі `internal/calculator.go` було реалізовано функції:

- `Add(a, b int) int` — додає два числа;
- `Divide(a, b int) (int, error)` — виконує ділення з перевіркою ділення на нуль;
- `IsEven(n int) bool` — перевіряє, чи є число парним.

Приклад реалізації:

```go
func Divide(a, b int) (int, error) {
	if b == 0 {
		return 0, errors.New("division by zero")
	}
	return a / b, nil
}
```

### 3. Написання unit-тестів

Для перевірки роботи функцій було створено файл `internal/calculator_test.go`. У тестах використано підхід **table-driven tests**. Було перевірено звичайні та граничні випадки: нульові значення, від’ємні числа та ділення на нуль.

Команда запуску тестів:

```powershell
go test -v ./...
```

### 4. Налаштування статичного аналізу коду

Було створено файл `.golangci.yml` з такими лінтерами:

```yaml
linters:
  enable:
    - errcheck
    - gosimple
    - unused
    - revive
    - staticcheck
```

Команда запуску лінтера:

```powershell
golangci-lint run
```

### 5. Демонстрація помилки, знайденої лінтером

Було спеціально створено ситуацію, коли помилка ігнорується:

```go
result, _ := internal.Divide(10, 2)
```

Після запуску `golangci-lint run` лінтер виявив проблему. Потім код було виправлено через явну обробку `error`.

### 6. Автоматизація через Makefile

Було створено `Makefile` з цілями:

```makefile
fmt:
	go fmt ./...

lint:
	golangci-lint run

test:
	go test -race -v ./...

build:
	go build -o bin/$(APP_NAME).exe ./cmd/app

all: fmt lint test build
```

Команда повної перевірки:

```powershell
make all
```

### 7. Налаштування `.gitignore`

Було створено `.gitignore`, у якому ігнорується директорія `bin/`, виконувані файли, тестові артефакти та службові файли IDE.

### 8. Робота з Git та GitHub

Проєкт було додано в Git-репозиторій:

```powershell
git init
git add .
git commit -m "Initial lab1 Go tooling setup"
```

## Висновок

У межах лабораторної роботи було налаштовано базове професійне середовище для розробки програм на Go. Використання `golangci-lint` дозволяє знаходити помилки ще до запуску програми, а `Makefile` спрощує виконання типових команд. Table-driven tests забезпечують зручний спосіб перевірки великої кількості вхідних даних, а GitHub Pull Request формує правильний підхід до командної розробки.

## Контрольні питання

### 1. Чим `golangci-lint` відрізняється від стандартного `go vet`?

`go vet` є стандартним інструментом Go, який шукає підозрілі конструкції у коді. `golangci-lint` є комплексним інструментом, який запускає багато лінтерів одночасно та дозволяє гнучко налаштовувати перевірки через конфігураційний файл.

### 2. Навіщо використовувати race detector під час тестування?

Race detector допомагає знаходити ситуації, коли кілька goroutine одночасно звертаються до одних і тих самих даних, і хоча б одна з них виконує запис.

### 3. Що таке `go.sum` і чому його не можна додавати в `.gitignore`?

`go.sum` містить контрольні суми модулів, які використовуються у проєкті. Він потрібен для перевірки цілісності залежностей.

### 4. Яка роль `Makefile`, якщо команди можна запускати вручну?

`Makefile` дозволяє стандартизувати типові дії: форматування, лінтинг, тестування та збірку.

### 5. Чому в Go прийнято повертати `error`, а не використовувати `panic/recover`?

Повернення `error` робить обробку помилок явною та передбачуваною. `panic/recover` краще використовувати лише для критичних ситуацій.
